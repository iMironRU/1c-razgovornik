## § 1.5 Контекст выполнения: клиент и сервер в одном файле

Это, пожалуй, самая непривычная конструкция для разработчика извне.

В веб-разработке клиент и сервер — это разные файлы, разные процессы, разные технологии. Граница между ними явная и физическая: HTTP-запрос. В 1С клиентский и серверный код может находиться в одном модуле формы, разделённый только **директивами компиляции** — аннотациями, которые говорят платформе, где должна выполняться каждая процедура.

```bsl
// Модуль формы — клиентский и серверный код рядом

&НаКлиенте
Процедура КнопкаНажата(Команда)
    // Выполняется у пользователя (в браузере или тонком клиенте)
    // Нет прямого доступа к базе данных
    Результат = ВычислитьНаСервере(100);
    Сообщить(Результат);
КонецПроцедуры

&НаСервере
Функция ВычислитьНаСервере(Значение)
    // Выполняется на сервере
    // Есть полный доступ к базе данных
    Возврат Значение * 2;
КонецФункции
```

Доступные директивы и их смысл:

| Директива | Где выполняется | Доступ к БД | Доступ к UI |
|---|---|---|---|
| `&НаКлиенте` | Клиент | Нет | Да |
| `&НаСервере` | Сервер | Да | Через реквизиты формы |
| `&НаСервереБезКонтекста` | Сервер | Да | Нет |
| `&НаКлиентеНаСервере` | Оба | Нет | Нет |

Для разработчика на React + Node.js аналогия такая: представьте, что компонент и API-обработчик написаны в одном файле, а аннотация `@client` или `@server` определяет, где что исполняется. Именно так работают директивы. Разница в том, что в 1С платформа сама организует вызов между контекстами — явный HTTP-запрос писать не нужно.

### Что можно передать через границу контекстов

При вызове серверной функции с клиента параметры сериализуются: передаются не как объекты в памяти, а как данные через внутренний канал платформы. Это означает, что не все типы можно передать — только **сериализуемые**.

Сериализуемые типы — те, которые платформа умеет однозначно упаковать и распаковать:

```
Строка, Число, Булево, Дата, Неопределено, Null
Массив         — если все элементы сериализуемы
Структура      — если все значения сериализуемы
Соответствие   — если все ключи и значения сериализуемы
СправочникСсылка, ДокументСсылка и другие ссылки — передаётся ID, не объект
```

Несериализуемые типы — те, которые содержат серверное состояние или произвольные вложенные типы:

```
ТаблицаЗначений    ← самая частая ловушка
ДеревоЗначений
СправочникОбъект, ДокументОбъект (объект целиком, не ссылка)
```

Таблица значений — мощный и часто используемый серверный тип. Передать её напрямую на клиент нельзя. Это не ошибка проектирования: таблица может содержать столбцы с произвольными типами, включая несериализуемые. Платформа не берётся сериализовать то, что не гарантированно сериализуемо.

Типичное решение — преобразовать таблицу значений в массив структур перед передачей:

```bsl
&НаСервере
Функция ПолучитьТоварыСервер()
    // Получаем таблицу значений из базы данных
    ТЗ = ПолучитьТаблицуТоваров();

    // Преобразуем в массив структур — это уже сериализуемо
    Результат = Новый Массив;
    Для Каждого Строка Из ТЗ Цикл
        Эл = Новый Структура("Наименование, Цена", Строка.Наименование, Строка.Цена);
        Результат.Добавить(Эл);
    КонецЦикла;

    Возврат Результат; // можно вернуть на клиент
КонецФункции
```

Для бэкенд-разработчика аналогия понятна: нельзя передать через REST-API сырой объект ORM — нужно сначала сериализовать его в JSON. Здесь та же идея, только граница проходит не между сервисами, а между контекстами одного приложения.

> Клиентский и серверный код в 1С может жить в одном файле — директивы определяют, где что выполняется. При передаче данных через эту границу работают только сериализуемые типы. Таблица значений через эту границу не проходит.

**Ловушки при переходе:**

*Из веб-разработки в 1С.* Инстинкт — думать о клиент-серверном взаимодействии как об HTTP. В 1С это внутренний канал платформы: он надёжнее, но работает по своим правилам сериализации.

*Из 1С в веб-разработку.* Инстинкт — ожидать, что типы данных «просто передаются» между клиентом и сервером. В REST-архитектуре всё проходит через JSON-сериализацию явно — нужно описывать структуру данных для передачи отдельно.
