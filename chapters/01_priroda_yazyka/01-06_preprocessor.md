## § 1.6 Директивы препроцессора

Помимо директив контекста выполнения, в 1С существуют **директивы препроцессора** — инструкции, которые обрабатываются до компиляции. Они начинаются с символа `#` и определяют, какой код вообще попадёт в компилятор.

```bsl
#Если Клиент Тогда
    // Этот блок будет скомпилирован только для клиентского контекста
    Процедура ОбновитьЭлементФормы()
        ЭлементыФормы.Кнопка.Доступность = Истина;
    КонецПроцедуры
#КонецЕсли

#Область СлужебныеПроцедуры
    // Область — только структурирование кода в редакторе
    // На компиляцию и выполнение не влияет никак
#КонецОбласти
```

Разработчику на C# это напоминает `#if DEBUG ... #endif` и `#region`. Аналогия точная: директива препроцессора — это не условие времени выполнения. Если условие ложно, соответствующий блок не попадёт в скомпилированный модуль вообще.

Важно не путать два разных механизма:

```
&НаКлиенте          — директива компиляции процедуры:
                      код скомпилирован, но выполняется только на клиенте

#Если Клиент Тогда  — директива препроцессора:
                      код не попадает в модуль вообще,
                      если условие не выполнено
```

На практике `#Если` чаще всего встречается в общих модулях, которые должны работать в разных контекстах, и в конфигурациях, поддерживающих несколько режимов работы. Для повседневного чтения кода достаточно понимать: всё, что внутри `#Если ... #КонецЕсли` — условно компилируемый блок.

> Директивы препроцессора (`#Если`) работают до компиляции — они определяют, какой код вообще попадёт в модуль. Директивы контекста (`&НаКлиенте`) работают при компиляции — они определяют, где скомпилированный код будет исполняться.

## Итог главы

Прежде чем 1С начнёт выглядеть знакомо, нужно принять несколько вещей.

Код в 1С описывает поведение уже существующей системы, а не строит систему с нуля. Архитектура платформы отражает устройство делового учёта: справочники — существительные, документы — глаголы, регистры — амбарная книга итогов. Вместо файлов и импортов — типы модулей и ключевое слово `Экспорт`. Типизация мягкая, но с конкретными правилами неявного приведения. Клиентский и серверный код могут жить в одном файле, разделённые директивами компиляции — но не все типы данных можно передать через эту границу.

Всё это — не случайные решения. Это последствия конкретной задачи: создать инструмент делового учёта, который работает на клиент-серверной архитектуре и понятен специалистам, говорящим по-русски. Знать эту логику полезно — она объясняет большинство мест, где 1С расходится с интуицией, натренированной на других языках.

Следующая глава — про переменные, типы данных и коллекции. Там начнётся синтаксис.
