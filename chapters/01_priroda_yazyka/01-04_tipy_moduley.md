## § 1.4 Типы модулей: вместо файлов и пакетов

В Python организация кода — это файлы и пакеты, связанные через `import`. В JavaScript — модули с `import/export`. В C# — файлы, пространства имён, сборки. В 1С нет ни того, ни другого в привычном виде.

Вместо этого существуют **типы модулей**. Каждый тип имеет своё назначение, свой контекст выполнения и свои правила видимости. Модуль не создаётся в файловой системе — он привязан к объекту конфигурации или существует как общий модуль.

```
Тип модуля              Где живёт             Назначение
──────────────────────  ────────────────────  ──────────────────────────────
Модуль приложения       Конфигурация          Старт и завершение сеанса
Модуль сеанса           Конфигурация          Инициализация параметров сеанса
Общий модуль            Конфигурация          Библиотека общих функций
Модуль объекта          Справочник/Документ   Логика конкретного объекта
Модуль менеджера        Справочник/Документ   Статические методы объекта
Модуль формы            Форма                 UI-логика и обработчики формы
```

Доступность кода между модулями определяется не `import`, а ключевым словом **`Экспорт`**. Функция без него видна только внутри своего модуля. Функция с `Экспорт` вызывается снаружи как `ИмяМодуля.ИмяФункции()`.

```bsl
// Общий модуль «РаботаСЦенами»

Функция ВнутренняяПроверка(Цена)   // видна только внутри модуля
    Возврат Цена > 0;
КонецФункции

Функция РассчитатьСкидку(Цена, Процент) Экспорт   // видна снаружи
    Возврат Цена * Процент / 100;
КонецФункции

// Вызов из любого другого места конфигурации:
// РаботаСЦенами.РассчитатьСкидку(1000, 10)
```

Общий модуль с признаком «глобальный» экспортирует свои функции без указания имени модуля — они доступны как будто встроены в язык. Это ближайший аналог `from module import *` в Python или глобального пространства имён в JavaScript.

Разработчику на C# структура напоминает разделение на статические классы-помощники, классы доменных объектов и контроллеры форм. Разница — отсутствие явного импорта: видимость определяется архитектурой конфигурации, а не директивами в начале файла.

> В 1С нет `import`. Видимость кода определяется типом модуля и ключевым словом `Экспорт`. Общие модули — ближайший аналог библиотек и статических классов.

**Ловушки при переходе:**

*Из Python/JS в 1С.* Инстинкт — искать `import` в начале файла, чтобы понять зависимости. В 1С его нет. Чтобы понять, откуда вызывается функция, нужно знать структуру конфигурации.

*Из 1С в Python/JS.* Инстинкт — писать «в одном месте» общую функцию и ожидать, что она будет видна везде. В Python это требует явного импорта в каждом файле, где функция используется.
